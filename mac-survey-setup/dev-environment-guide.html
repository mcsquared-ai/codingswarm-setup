Clean Dev Environment Guide

For Claude Code · GCP Sandbox · Enterprise AI Systems of Action
Prepared for: Pankaj Shroff, mcsquared.ai · March 2026

ℹ️ Audit Scope Note
This guide was produced after inspecting your setup context and running diagnostic tooling. For a live audit of your current Mac state (to catch existing conflicts before you install), open Cowork, click Select Folder and point it at your home directory (~), then ask to re-run the audit. Everything below is correct practice regardless of current state — apply it as a migration or a fresh start.

The goal is a single source of truth for every runtime, binary, and dependency on your machine — one tool manages Python versions, one manages Node versions, one manages system binaries, cloud CLIs use official installers, and every project carries its own isolated environment. No global pip installs. No Homebrew Python used as project Python. No competing PATH entries.

Your use case — building intelligent systems of action that are portable across GCP, AWS, and Azure, and that connect to enterprise Systems of Record (Snowflake, Databricks, BigQuery, Palantir Foundry, Oracle) — means your local environment also needs to be your integration test harness. Getting this right upfront is the difference between compounding productivity and compounding tech debt.

Anti-Patterns That Will Break You

❌ Never Do These
  • pip install pandas without an active venv
  • brew install python then using it as project Python
  • Mixing conda environments with pip/venv
  • Hardcoded credentials or API keys in code
  • Committing .env files (even "empty" ones)
  • Global Node packages that should be project deps
  • Multiple pyenv and asdf both active
  • Installing CLI tools via pip install --user
  • Dev/beta versions of libraries in production paths
✅ Always Do These
  • One Python version manager (pyenv), zero exceptions
  • One venv per project in a consistent location
  • .python-version file in every project root
  • .nvmrc file in every project root
  • .env.example committed; .env gitignored
  • Pin library versions: requirements.lock or uv.lock
  • ADC (Application Default Credentials) for GCP auth
  • Test with GCP sandbox before any client deployment
  • CLAUDE.md in every project for Claude Code context

1 · Shell & Terminal Setup

Install iTerm2 + Starship

# iTerm2 — better terminal
brew install --cask iterm2

# Starship prompt — fast, context-aware, shows venv/git/gcloud/python versions
brew install starship

# Add to end of ~/.zshrc:
eval "$(starship init zsh)"

Starship config (~/.config/starship.toml)

This config shows: directory, git branch, Python venv, Python version, GCP project, Node version, and exit codes — everything you need at a glance.

# ~/.config/starship.toml
# Shows: dir · git · python venv · python version · gcp project · node · status

format = """
[$directory](bold blue)\
$git_branch\
$git_status\
$python\
$gcloud\
$nodejs\
$cmd_duration\
$line_break\
$character"""

[directory]
truncation_length = 4
truncate_to_repo = true

[python]
format = '[(\($virtualenv\) )](italic #f5a623)[${symbol}${pyenv_prefix}(${version})]($style) '
symbol = "🐍 "
detect_files = ["pyproject.toml", "requirements.txt", ".python-version", "setup.py"]

[gcloud]
format = '[$symbol$account(@$domain)(\($region\))]($style) '
style = "bold blue"
symbol = "☁️  "

[git_branch]
format = '[$symbol$branch]($style) '
style = "bold purple"

[character]
success_symbol = "[❯](bold green)"
error_symbol = "[❯](bold red)"

Key iTerm2 Profile Settings

Preferences → Profiles → General: Set working directory to "Reuse previous session's directory"

Preferences → Profiles → Colors: Use Catppuccin Mocha or Dracula (avoid default white, reduces eye strain for long sessions)

Preferences → Profiles → Keys: Enable natural text editing (Option+← to jump words)

Preferences → General → Magic: Enable Shell Integration — this gives you badge showing active venv name

📁 The .envs/ pattern
CLI tools like dbt, databricks-cli, and snowflake-connector-python need Python but shouldn't live in any project's venv. Put them in a shared ~/.envs/data-tools/ venv, activate it when needed, and alias it. This keeps project venvs lean and CLI tools stable.

3 · Homebrew — System Binaries Only

⚠️ Golden Rule
Homebrew manages system-level CLI tools and compilers. It does NOT manage Python versions you use for projects. It does NOT manage Node.js versions you use for projects. Those are handled by pyenv and nvm respectively.

4 · Python — pyenv + venv

Install & Configure pyenv

# Install pyenv (via brew — this is correct)
brew install pyenv

# Add to ~/.zshrc (see template file for full config):
export PYENV_ROOT="$HOME/.pyenv"
export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
eval "$(pyenv virtualenv-init -)"  # Only if using pyenv-virtualenv plugin

Install Python Versions

# Install build dependencies first (critical on macOS)
brew install openssl readline sqlite3 xz zlib tcl-tk

# Install Python 3.12 (latest stable — use for new projects)
pyenv install 3.12.3

# Install Python 3.11 (LTS — use for production workloads with proven libs)
pyenv install 3.11.9

# Set global Python (used when no project-level .python-version exists)
pyenv global 3.12.3

# Verify
python --version      # Should show 3.12.3
which python          # Should show ~/.pyenv/shims/python
which pip             # Should show ~/.pyenv/shims/pip
🚫 Never Do This
pip install pandas without an active virtual environment. If you run which pip and it shows /usr/local/bin/pip or /opt/homebrew/bin/pip — stop. You are about to pollute your global Python. Activate a venv first.

Python Package Management — Two Approaches

Option A: uv (Recommended for New Projects)

uv is a next-generation Python package manager (by Astral, the makers of ruff). It's 10–100x faster than pip, handles lockfiles, and is designed for professional workflows.

# Install uv
pip install uv  # Or: curl -LsSf https://astral.sh/uv/install.sh | sh

# New project with uv
mkdir ~/dev/internal/my-agent && cd ~/dev/internal/my-agent
uv init                             # Creates pyproject.toml, .python-version, .venv
uv add anthropic google-cloud-bigquery snowflake-connector-python
uv add --dev pytest ruff mypy

# Run commands inside project venv without activating
uv run python main.py
uv run pytest

# Lock dependencies for reproducibility
uv lock                             # Creates uv.lock — commit this to git
uv sync                             # Installs from lockfile on another machine

Option B: pip + venv (Classic, More Universal)

# Every new project — do this once
cd ~/dev/internal/my-agent
pyenv local 3.12.3                  # Creates .python-version file
python -m venv .venv                # Creates .venv/ inside project
source .venv/bin/activate           # Activate (or let direnv do this)

# Install packages — ONLY when venv is active (you'll see (.venv) in prompt)
pip install anthropic google-cloud-bigquery snowflake-connector-python
pip install --upgrade pip

# Freeze for reproducibility
pip freeze > requirements.lock      # Exact versions — use for deployment

Shared Venvs for CLI Tools (in ~/.envs/)

# Create shared data tools environment
python -m venv ~/.envs/data-tools
source ~/.envs/data-tools/bin/activate
pip install dbt-core dbt-bigquery databricks-cli snowflake-connector-python
deactivate

# Add shell aliases (in .zshrc)
alias data-tools="source ~/.envs/data-tools/bin/activate"
alias gcp-tools="source ~/.envs/gcp-tools/bin/activate"

5 · Node.js — nvm

# Install nvm (via brew is fine)
brew install nvm

# Add to ~/.zshrc:
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion"

# Install Node LTS (use for all new projects)
nvm install --lts
nvm use --lts
nvm alias default 'lts/*'

# Per-project pin (.nvmrc file)
echo "20" > .nvmrc          # Node 20 LTS
nvm use                     # Reads .nvmrc automatically

# Auto-use .nvmrc when entering directory (add to .zshrc)
autoload -U add-zsh-hook
load-nvmrc() {
  local nvmrc_path="$(nvm_find_nvmrc)"
  if [ -n "$nvmrc_path" ]; then
    local nvmrc_node_version=$(nvm version "$(cat "${nvmrc_path}")")
    if [ "$nvmrc_node_version" = "N/A" ]; then
      nvm install
    elif [ "$nvmrc_node_version" != "$(nvm version)" ]; then
      nvm use
    fi
  fi
}
add-zsh-hook chpwd load-nvmrc
load-nvmrc

6 · GCP — gcloud CLI

📌 Install via Official Installer — Not Homebrew
The official installer bundles its own Python, manages its own updates, and doesn't conflict with your system Python. Always use it. 
# Download and install
curl https://sdk.cloud.google.com | bash
# Restart shell after install
exec -l $SHELL

# Initialize and auth
gcloud init
gcloud auth login
gcloud auth application-default login  # ADC — use this for all SDK auth

# Set your sandbox project as default
gcloud config set project YOUR_GCP_PROJECT_ID
gcloud config set compute/region us-central1
gcloud config set compute/zone us-central1-a

# Create named config for sandbox vs production
gcloud config configurations create sandbox
gcloud config set project YOUR_SANDBOX_PROJECT_ID
gcloud config configurations create production
gcloud config set project YOUR_PROD_PROJECT_ID

# Switch between configs
gcloud config configurations activate sandbox
gcloud config configurations activate production

# Enable APIs you'll need (run in sandbox project)
gcloud services enable \
  aiplatform.googleapis.com \
  bigquery.googleapis.com \
  run.googleapis.com \
  cloudbuild.googleapis.com \
  secretmanager.googleapis.com \
  storage.googleapis.com \
  compute.googleapis.com \
  container.googleapis.com \
  artifactregistry.googleapis.com

# Verify ADC is working
gcloud auth application-default print-access-token

GCP Authentication in Code

# In Python code — NEVER hardcode credentials
# ADC handles auth automatically when you've run gcloud auth application-default login

from google.cloud import bigquery
from google.cloud import storage

# ADC is used automatically — no credentials argument needed in local dev
client = bigquery.Client(project="your-project-id")

# In .env for local dev
GOOGLE_CLOUD_PROJECT=your-sandbox-project-id
# NEVER put GOOGLE_APPLICATION_CREDENTIALS in code or .env for dev
# Use ADC (application-default login) instead

Service Account Pattern for CI/CD & Production

# Create service account for each workload
gcloud iam service-accounts create my-service \
  --display-name="My Service Account"

# Grant minimal permissions (principle of least privilege)
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:my-service@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/bigquery.dataViewer"

# For Cloud Run — no key files needed, use Workload Identity
# For local testing of production SA — use impersonation
gcloud auth application-default login \
  --impersonate-service-account=my-service@PROJECT_ID.iam.gserviceaccount.com
